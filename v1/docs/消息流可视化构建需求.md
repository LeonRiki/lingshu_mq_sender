# 消息流可视化构建需求

## 目标

在测试用例的 `input` 消息构建区直接展示本次消息流。用户以消息气泡和发送间隔编排消息，系统根据发送间隔自动计算最终发送的 JSON 请求，无需手动选择“单次发送”或“多次发送”。

## 核心概念

| 元素 | 含义 | 对 JSON 的影响 |
| --- | --- | --- |
| 消息气泡 | 一条待发送的用户消息 | 与同一批次内的其他消息共同写入一个 JSON 的 `input` 数组 |
| 发送间隔 | 两个发送批次之间的等待时间 | 将间隔前、后的消息拆分为两个 JSON 请求 |

消息序列中连续的消息气泡属于同一个发送批次；遇到发送间隔时结束当前批次并开始下一批次。

示例：

```text
消息 A -> 消息 B -> 等待 3 秒 -> 消息 C
```

执行结果为：

```text
JSON 1: { input: ["消息 A", "消息 B"] }
等待 3 秒
JSON 2: { input: ["消息 C"] }
```

## 交互需求

1. 构建区展示已确认的消息气泡与发送间隔行，并在底部提供“添加消息”和“添加发送间隔”。
2. 点击“添加消息”后展示输入框；填写并确认后，生成一条消息气泡。取消不产生消息。
3. 点击消息气泡的编辑操作可修改内容；删除消息后，系统自动清理因此变得无效的相邻间隔。
4. “添加发送间隔”仅在至少有两条消息时可用，新增的间隔默认插入最后两条消息之间，默认值为 3 秒。
5. 发送间隔可修改秒数、删除，并可通过拖拽在消息流中移动。
6. 拖拽后若会产生开头间隔、末尾间隔或连续间隔，系统不接受该次移动。
7. 消息流必须以消息气泡开始和结束；不允许连续两个发送间隔。
8. 消息预览展示与构建区一致的气泡流，并明确标出每个等待节点。
9. JSON 模式展示系统计算出的最终请求：没有间隔时为一个 JSON；存在间隔时为 JSON 数组。

## 执行与数据规则

1. 发送模式由消息流自动推导，不在编辑界面提供“单次发送/多次发送”开关。
2. 没有发送间隔时，所有消息气泡合并为一个 JSON 并发送一次。
3. 每个有效发送间隔额外形成一个 JSON 请求；若有 `n` 个间隔，则发送 `n + 1` 次。
4. 后续 JSON 的 `inputList` 包含此前已发送批次中的全部用户消息，以保持原有会话历史语义。
5. 用例持久化采用 `conversation.flow`，其中元素为 `message` 或 `delay`。服务端读取旧版 `conversation.messages + intervalSeconds` 用例时转换为等价消息流，保证已有用例可继续发送。
6. `meta.mode` 为兼容旧记录保留：无间隔时为 `single`，有间隔时为 `multi`；它不再是用户可编辑的配置。

## 验收标准

1. 连续添加三条消息且不加间隔，发送记录中只有一条 MQ 消息，`input` 包含三项。
2. 在三条消息的第二、第三条之间添加 2 秒间隔，发送记录中有两条 MQ 消息：第一条 `input` 含前两项，第二条仅含第三项，第二条等待 2 秒。
3. 拖动间隔到另一组相邻消息之间后，JSON 分组与发送记录随之变化。
4. 不能保存或发送开头/末尾间隔、连续间隔或空消息。
5. 旧版单次、多次用例打开后可正常显示、保存与发送。
